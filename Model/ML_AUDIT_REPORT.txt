1.1 Active Architecture (pooling="stats_direct")
    ┌─────────────────────────────────────────────────────────────┐
    │ Input: (B, T, 384) — SBERT text embeddings per chunk       │
    │   ↓                                                        │
    │ StatisticsPooling(mean+std) → (B, 768)    [0 params]       │
    │   ↓                                                        │
    │ BatchNorm1d(768) → (B, 768)               [1,536 params]   │
    │   ↓                                                        │
    │ Linear(768, 1) → (B,)                     [769 params]     │
    │   ↓                                                        │
    │ Output: PHQ-8 score (scalar)                               │
    └─────────────────────────────────────────────────────────────┘
    Total: 2,305 learnable parameters

1.2 Statistics Pooling (src/models/statistics_pooling.py)
    - Mode "mean_std": concatenates per-feature mean and std → D×2 = 768
    - Mode "mean_std_min_max" (unused): D×4 = 1,536
    - Mask-aware: uses Bessel-corrected std, handles variable-length sequences
    - Zero learnable parameters — acts as a deterministic summary operator
    VERDICT: ✓ Correct implementation. Mean+std is appropriate for capturing
    distributional properties of text embeddings across speech chunks.

1.3 Inactive Architecture Components (intact but bypassed)
    All components below exist in the codebase but are NOT used in training
    when pooling="stats_direct":
    - MLPBlock (mlp_block.py): 2-layer bottleneck with weight norm, GELU,
      LayerNorm. Would be used in pooling="stats" or pooling="attention".
    - BiGRUEncoder (bigru.py): Bidirectional GRU with pack_padded_sequence,
      LayerNorm, dropout. Would model temporal dependencies.
    - AttentionPooling (attention.py): Additive attention with entropy
      tracking. Used with pooling="attention". Previously caused mean collapse.
    - GatedSkipConnection: MLP mean-pool → projection + gated fusion with
      attention output. Used with pooling="attention".
    VERDICT: ⚠ Dead code. The active model discards ALL temporal modeling.
    The architecture is functionally a linear regression on pooled statistics.

1.4 Critical Architecture Finding
    The active model (stats_direct + Linear head) is mathematically equivalent
    to:
        PHQ = w · [mean(text); std(text)] + b
    after BatchNorm absorbs the centering/scaling. This is a Ridge regression
    with learned regularization (via BatchNorm + weight_decay), not a deep
    learning model. The "neural network" framing is misleading for a 2,305-
    parameter linear map.

2.2 Text Feature Pipeline (active path: feature_mode="text_only")
    Raw audio → Whisper ASR → transcript segments → SBERT encoding → L2 norm
    - Whisper transcribes each audio chunk into text
    - SBERT (all-MiniLM-L6-v2, 384-dim) encodes each segment
    - L2 normalization per chunk: text / ||text||₂
    - No further processing — raw L2-normed embeddings go to the model

2.3 Audio Feature Pipeline (BYPASSED in current config)
    - eGeMAPS(88): StandardScaler (fit on train, transform all)
    - MFCC(120): StandardScaler (fit on train, transform all)
    - Feature fusion: concat [eGeMAPS|MFCC|text] → 592 dims
    - PCA reduction: 592 → 24 dims (or configurable n_components)
    VERDICT: ⚠ Audio features are still EXTRACTED during data preparation
    (275 feature files × 3 modalities) but are completely IGNORED during
    training and inference. This wastes disk space and extraction compute.

2.4 Critical Feature Finding
    Empirical evidence from Ridge regression baselines:
    ┌──────────────────────────────────────────────────────────────┐
    │ Feature Set                  │ Ridge CCC │ α       │ Dims   │
    ├──────────────────────────────┼───────────┼─────────┼────────┤

    The text modality carries >95% of the depression signal. Audio features

2.5 Feature Quality Issues
    [ISSUE-F1] SBERT dependency: The all-MiniLM-L6-v2 model is a general-
        purpose sentence encoder, not fine-tuned for clinical/depression
        language. Task-specific fine-tuning could improve embeddings.
    [ISSUE-F2] ASR bottleneck: Whisper transcription errors propagate into
        SBERT embeddings. No ASR error correction or confidence-based
        filtering is applied. avg_logprob from Whisper is extracted but
        completely ignored.
    [ISSUE-F3] eGeMAPS/MFCC waste: 208 audio features are extracted per
        chunk (85 chunks × 208 dims for a typical subject) but never used.
        Feature extraction time includes openSMILE and librosa processing
        that serves no purpose.

3.2 Loss Function: WeightedMSELoss
    L = (1/N) Σ wᵢ · (predᵢ - targetᵢ)²
    where wᵢ = 2.0 if PHQ ≥ 10, else 1.0

    VERDICT: ✓ Addresses class imbalance with binary severity weighting.
    CCC was tested as a training loss but caused unstable gradients due to
    small batch statistics (CCC from <20 samples is unreliable). CCC is
    correctly used only as the EVALUATION metric for checkpointing/early
    stopping.

3.3 Class Imbalance Handling
    Two mechanisms (potentially conflicting):
    1. WeightedMSELoss: high_weight=2.0 for PHQ≥10, low_weight=1.0
    2. WeightedRandomSampler: inverse-frequency sampling (depressed 30.7%
       → sampled at higher rate, replacement=True)
    [ISSUE-T1] With batch_size=163 (full dataset), the sampler draws 163
        samples WITH replacement from 163 subjects. On average, ~37% of
        subjects will be MISSING each epoch (1-(1-1/163)^163 ≈ 63% seen).
        This is effectively bootstrap sampling, NOT balanced sampling.
        Combined with the weighted loss, the class rebalancing is applied
        TWICE with different strategies — this creates unpredictable
        effective weights.

5.1 Train/Val Loss Gap
    At best epoch (145):
        Train loss: 1.6580
        Val loss:   29.9884
        Ratio:      18.1× (val is 18× train)

    This is a MASSIVE gap. The model memorizes training data (weighted MSE
    near zero on seen subjects) but generalizes poorly in terms of raw MSE.

5.2 Why CCC Is High Despite High Val Loss
    CCC measures correlation agreement, not absolute error. The model has
    learned the correct RANKING of subjects by depression severity (Pearson
    ρ ≈ 0.58) with acceptable mean/variance calibration, but the absolute
    squared errors are large (RMSE = 4.56 on a 0–24 scale).

    This means:
    - The model roughly knows who is more depressed than whom (good CCC)
    - Individual predictions are off by ~3.5 PHQ points on average (bad MAE)
    - For clinical use (screening), ranking may suffice
    - For clinical use (severity assessment), MAE=3.5 is insufficient

5.3 Parameter-to-Sample Ratio
    - Total parameters: 2,305
    - Training samples: 163
    - Ratio: 163 / 2,305 = 0.07 (samples per parameter)
    - Convention: need ≥5–10 samples per parameter for reliable learning
    VERDICT: ✗ SEVERELY under-sampled. 14:1 ratio IS reported in the summary
    file, but this counts the inverse — you need 2,305 parameters / 163
    samples = each parameter sees <1 unique sample. The BatchNorm1d(768)
    alone has 1,536 parameters (running mean + running var + weight + bias)
    learning from 163 samples, meaning the normalization statistics are
    fragile.

    However: BatchNorm running_mean/running_var are not gradient-trained
    (updated via momentum), and weight/bias are 768 each → effectively
    768 learnable BN params + 769 linear params = 1,537 gradient-trained
    params. Ratio: 163/1,537 ≈ 0.11 samples/param. Still poor.

    The model works DESPITE this ratio because:
    1. Text embeddings are pre-trained (SBERT carries all the knowledge)
    2. Mean+std pooling is a deterministic summary (no learned pooling)
    3. The effective task is just "learn 768 weights for a linear map"
    4. Weight decay 0.1 provides strong L2 regularization

5.4 WeightedRandomSampler Overfitting Interaction
    With replacement=True on 163 subjects, each epoch sees ~103 unique
    subjects (bootstrap). Some subjects are sampled 2–4 times per epoch.
    After 145 epochs, the model has "memorized" the duplicated subjects
    while never seeing ~37% of subjects in any given epoch. This creates
    uneven optimization across the training set and inflates train loss
    variance.

5.5 Overfitting Verdict
    OVERFITTING SEVERITY: MODERATE
    - Train/val loss ratio is extreme (18×) → classic overfitting signal
    - BUT CCC remains stable (0.54 at best, 0.52 at last epoch) → the
      model's ranking ability is NOT degrading
    - The LR scheduler reduced LR to 1e-6 → training effectively stopped
      long before epoch 445
    - Weight decay 0.1 acts as strong regularization on the linear layer
    - 300-epoch patience is excessive: the model converged by epoch ~150

6.3 Capacity Utilization
    Effective degrees of freedom: 768 (linear weights) + 1 (bias) = 769
    BatchNorm weight/bias are scaling/shifting, not independent features.
    The model is learning: "which of the 768 mean/std text features
    correlate positively or negatively with depression severity?"
    This is a well-posed problem for N=163 if the features are
    pre-normalized (they are, via L2 norm + BatchNorm).

[CRITICAL-4] DOUBLE CLASS REBALANCING CONFLICT
    Severity: ■■■□□ (3/5)
    The training pipeline applies TWO independent rebalancing strategies:
    1. WeightedRandomSampler (oversamples depressed subjects)
    2. WeightedMSELoss (2× weight for depressed subjects)
    These compound unpredictably. With full-batch training, the sampler
    produces bootstrap samples (~37% subjects missing per epoch), and the
    loss adds another 2× multiplicative weight on depressed subjects.
    Effective weight on depressed subjects ≈ 2× (loss) × 1.6× (sampler) ≈
    3.2× instead of the intended 2×.
    FIX: Remove WeightedRandomSampler. With batch_size=163 (full dataset),
    use shuffle=True instead and let the weighted loss handle imbalance.

9.2 Multimodal Fusion Fix (Expected Gain: +0.05–0.15 CCC)
    The current early fusion (concat → PCA) fails because PCA destroys
    the text signal. Alternatives:
    a) Late fusion: Train separate text-only and audio-only heads, then
       learn a weighted combination of their predictions. This prevents
       audio noise from contaminating text features.
       Implementation:
           text_pred = Linear(text_mean_std)      # CCC ≈ 0.54
           audio_pred = Linear(audio_mean_std)     # CCC ≈ 0.12
           final = α * text_pred + (1-α) * audio_pred  # Learn α
    b) Cross-modal attention: Use text as query, audio as keys/values.
       Let the model learn which audio frames are relevant given the
       text context. Requires more data than currently available.
    c) Replace eGeMAPS/MFCC with wav2vec 2.0 or HuBERT features:
       - Self-supervised audio representations capture much richer
         information than handcrafted features
       - wav2vec 2.0 features have shown CCC > 0.5 for audio-only
         depression detection in published work
       - Combined with text, could reach CCC 0.65+

9.3 Architecture Upgrades (Expected Gain: +0.03–0.10 CCC)
    a) Temporal modeling: Re-enable BiGRU path but on text-only features.
       The current stats_direct ignores temporal patterns in speech. The
       ORDER of chunks carries information (depression may manifest more
       in certain conversation phases).
    b) Attention over temporal statistics: Compute statistics per time
       window (e.g., 5 windows), then attend over windows to capture
       temporal dynamics without full sequence modeling.
    c) Pre-trained transformer backbone: Use a pre-trained model like
       Whisper encoder or wav2vec as the feature extractor, fine-tuned
       end-to-end with a PHQ regression head.

9.4 Training Upgrades (Expected Gain: +0.02–0.05 CCC)
    a) Multi-task learning: Predict both PHQ-8 (regression) and PHQ
       binary (classification, ≥10) simultaneously. The binary task
       provides cleaner gradients, especially for borderline subjects.
    b) Data augmentation: Mixup (interpolate between subjects' pooled
       features and labels). More suitable than temporal augmentation
       for the current pooling architecture.
    c) Ensemble: Train 5 models with different seeds, average predictions.
       Cheap ensembling typically gives +0.02–0.03 CCC at inference cost.
