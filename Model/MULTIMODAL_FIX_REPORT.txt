1. ROOT CAUSE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CURRENT STATE:
  Text-only CCC:     0.54  (model: StatsPool(mean+std, 384→768) → BN → Linear → 1)
  Multimodal CCC:    0.13  (fused 592-dim with PCA → 24-dim)
  Audio acting as:   NOISE (destroying text signal)

ROOT CAUSES (5 failures identified):

  1. MFCC + eGeMAPS ARE WEAK FEATURES FOR THIS TASK
     ─────────────────────────────────────────────────
     eGeMAPS (88-dim):  Hand-crafted functionals (means, percentiles over segment).
                        Good for paralinguistic tasks with BIG datasets (1000+ subjects).
                        With 163 training subjects, 88 eGeMAPS dims → p >> n problem.
                        These features average away the temporal patterns
                        (pauses, speech rate changes, vocal tremor) that actually
                        correlate with depression.

     MFCC (120-dim):    40 MFCCs + delta + delta-delta, mean-pooled per segment.
                        MFCCs capture spectral envelope — useful for ASR, not for
                        paralinguistic/affect tasks. Mean-pooling over time discards
                        prosodic dynamics entirely. The 120 dimensions are mostly
                        spectral detail noise with no depression signal.

     Combined (208 audio dims vs 384 text dims):
       Audio features have LOW signal-to-noise ratio for depression.
       When fused with high-signal text (384-dim), the 208 noisy audio dims
       DILUTE the text signal. PCA on 592-dim to 24-dim mixes modalities
       and the strong text components get corrupted by audio noise.

  2. PCA MIXING DESTROYS MODALITY BOUNDARIES
     ─────────────────────────────────────────
     Current pipeline: [eGeMAPS(88) ⊕ MFCC(120) ⊕ Text(384)] → PCA(592→24)

     PCA is a linear rotation. It finds principal components across ALL 592 dims.
     Since audio features have high variance but LOW correlation with PHQ-8,
     the top PCA components capture audio variance (spectral energy variation)
     rather than text→depression signal. This is the classic "dominant noise
     modality" problem in early fusion.

     Result: PCA-reduced 24-dim features are mostly audio variance with
     trace text signal → CCC collapses from 0.54 to 0.13.

  3. TRAIN-INFERENCE SEGMENTATION MISMATCH
     ──────────────────────────────────────
     Training:   Variable-length segments (0.3s–57s) from transcript CSV
     Inference:  Fixed 5.0s windows, 25% overlap

     This creates a distribution shift:
       - Training segments have variable statistics (a 0.3s "yeah" has very
         different eGeMAPS than a 30s monologue)
       - Inference segments are uniformly 5s → different feature distributions
       - Statistics pooling (mean+std over segments) computes fundamentally
         different values between training and inference
       - BatchNorm statistics fitted on training distributions are WRONG for
         the inference segment distribution

  4. NAIVE CONCATENATION FUSION HAS NO QUALITY GATING
     ─────────────────────────────────────────────────
     Current fusion: np.concatenate([egemaps, mfcc, text], axis=1)

     Problems:
       - No mechanism to suppress audio when it's unreliable
       - Silent segments produce meaningless eGeMAPS/MFCC → noise injected
       - Model cannot LEARN to ignore audio — it's pre-fused before the model

  5. NO STAGED TRAINING — AUDIO OVERFITS IMMEDIATELY
     ─────────────────────────────────────────────────
     Training all parameters jointly from random init means:
       - Audio branch fits training noise faster than text (208 free dims)
       - Text signal gets diluted by audio gradients
       - With 163 subjects, joint optimization doesn't have enough data
         to learn modality importance without explicit guidance

2. AUDIO FEATURE UPGRADE — HuBERT Base
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  REPLACEMENT:
    OLD: eGeMAPS(88) + MFCC(120) = 208 hand-crafted dims
    NEW: HuBERT Base(768) = 768 learned dims

  WHY HuBERT > MFCC + eGeMAPS:
    ┌────────────────────┬─────────────────────┬─────────────────────────┐
    │ Aspect             │ MFCC/eGeMAPS        │ HuBERT Base             │
    ├────────────────────┼─────────────────────┼─────────────────────────┤
    │ Features           │ Hand-crafted        │ Learned from 960h speech│
    │ Captures prosody   │ Partially (F0 only) │ Yes (implicit in repr)  │
    │ Captures pauses    │ No (mean-pooled)    │ Yes (temporal modeling)  │
    │ Noise robustness   │ Low                 │ High (pre-trained)      │
    │ Dims               │ 208 (fragmented)    │ 768 (unified)           │
    │ Prior on speech    │ None                │ 960h unsupervised       │
    │ SER literature     │ Baseline            │ State-of-the-art        │
    └────────────────────┴─────────────────────┴─────────────────────────┘

    Key insight: HuBERT's self-supervised pre-training learns representations
    that encode speaker characteristics, prosody, speaking rate, and vocal
    quality — ALL clinically validated depression markers. These features
    come "for free" without being hand-designed for depression.

    Literature: HuBERT/wav2vec2 features achieve CCC 0.3-0.5 for audio-only
    depression detection on DAIC-WOZ (vs CCC 0.1-0.2 for eGeMAPS).

  HOW TO EXTRACT:
    1. Load pre-trained facebook/hubert-base-ls960 (frozen, ~360MB)
    2. For each audio segment: pass through HuBERT → get last hidden layer
    3. Mean-pool over time frames → one 768-dim vector per segment
    4. No fine-tuning of HuBERT itself (frozen feature extractor)

  IMPLEMENTATION: src/features/hubert_extractor.py
    - HuBERTExtractor class with lazy model loading
    - extract_single(audio_segment) → (768,)
    - extract_from_audio(segments_list) → (N, 768)
    - GPU-accelerated, handles short segments by padding to 25ms minimum
    - Fits in 6GB VRAM (RTX 4050) alongside SBERT

3. SEGMENTATION FIX — Fixed 5s Windows for Training
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  DECISION: Option A — Train on fixed 5s windows (matching inference)

  JUSTIFICATION:
    Option A (fixed windows for training):
      ✓ Eliminates train-inference distribution shift completely
      ✓ Uses Whisper for text (same as inference) → consistent text quality
      ✓ Fixed-length segments produce more stable audio features
      ✓ Overlapping windows increase effective training samples

    Option B (transcript-based inference):
      ✗ Requires ASR + forced alignment at inference → added complexity & latency
      ✗ Inference user may not have transcript
      ✗ Variable-length segments cause feature instability with small models

    Option A is strictly better for production systems.

  IMPLEMENTATION: scripts/extract_features_v2.py
    - Fixed 5.0s chunks, 25% overlap, 3.75s hop (identical to AudioChunker)
    - Whisper Base transcription per chunk → SBERT encoding
    - HuBERT extraction per chunk
    - Audio quality scoring per chunk
    - Output: data/features_v2/{pid}_training_v2.npz

5. AUDIO QUALITY SCORING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  DESIGN: src/features/audio_quality.py → AudioQualityScorer

  Metrics per segment:
    ┌───────────────────┬──────────────────────────────────────────┐
    │ Metric            │ What it measures                         │
    ├───────────────────┼──────────────────────────────────────────┤
    │ SNR (0.4 weight)  │ Signal vs noise ratio. Low SNR → noisy   │
    │                   │ audio → unreliable features.             │
    │ RMS Energy (0.3)  │ Overall signal level. Near-zero → silent │
    │                   │ segment. Sigmoid-normalized.             │
    │ Speech Prob (0.3) │ Fraction of frames above silence thresh. │
    │                   │ Low → interviewer silence / dead air.    │
    └───────────────────┴──────────────────────────────────────────┘

  Output: quality ∈ [0, 1] per segment, stored in NPZ.

  How it's used:
    Model has a QUALITY GATE layer:
      quality_gate(quality_score) → sigmoid → [0, 1]
      audio_features *= quality_gate_output

    If quality is low   → gate ≈ 0 → audio features zeroed → text dominates
    If quality is high   → gate ≈ 1 → audio features pass through → fusion

    This is DYNAMIC per-segment — learned end-to-end.

6. FUSION CORRECTION — Gated Multimodal Fusion
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  OLD FUSION:
    [eGeMAPS(88) ⊕ MFCC(120) ⊕ Text(384)] → PCA(592→24) → Model
    Problems: no quality gating, PCA mixes modalities, audio dilutes text.

  NEW FUSION: src/models/gated_fusion_model.py → GatedMultimodalModel

  Architecture:
    ┌──────────────────────────────────────────────────────────────┐
    │                                                              │
    │  Text(B,T,384)──→ StatsPool(mean+std) → BN → Linear → 128  │
    │                                                    ↓         │
    │  Audio(B,T,768)→ QualityGate → StatsPool → BN → Linear→128 │
    │                                                    ↓         │
    │           ┌─────────────────────────────────────────┐        │
    │           │ Fusion Gate: [text_proj ⊕ audio_proj]   │        │
    │           │ → MLP → Sigmoid → gate g ∈ [0,1]       │        │
    │           │                                         │        │
    │           │ fused = g * text + (1-g) * audio        │        │
    │           └─────────────────────────────────────────┘        │
    │                          ↓                                   │
    │                   Linear(128, 1)                             │
    │                          ↓                                   │
    │                     PHQ-8 score                              │
    └──────────────────────────────────────────────────────────────┘

  KEY SAFETY MECHANISMS:
    1. Quality Gate: per-segment, suppresses bad audio BEFORE pooling
    2. Fusion Gate:  learned, can learn g→1 to use text-only if audio hurts
    3. Modality Dropout: 15% chance of zeroing audio during training
       → model MUST learn to function with text-only as fallback
    4. Separate branches: no PCA mixing, each modality has its own projection

  PARAMETER BUDGET:
    Text branch:    768 → 128 = 98,560 params
    Audio branch:   1536 → 128 = 196,992 params
    Quality gate:   2 params
    Fusion gate:    128*2 → 128 → 1 = 33,025 params
    Head:           128 → 1 = 129 params
    BN layers:      768*2 + 1536*2 = 4,608 params
    ─────────────────────────────────────────────
    Total:          ~333K params
    Ratio:          163 subjects / 333K params → requires staged training

7. TRAINING STRATEGY — Three-Stage Staged Training
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  IMPLEMENTATION: scripts/train_multimodal.py

  ┌─────────────────────────────────────────────────────────────────┐
  │ Stage 1: TEXT-ONLY BASELINE                                     │
  │ ─────────────────────────────────                               │
  │ Freeze: audio branch, fusion gate                               │
  │ Train:  text branch + head                                      │
  │ LR:     1e-2, WD: 0.1                                          │
  │ Goal:   Establish text CCC ≥ 0.50                               │
  │ Why:    Text is the strong modality. Train it first so audio    │
  │         cannot degrade it. This is the floor performance.       │
  ├─────────────────────────────────────────────────────────────────┤
  │ Stage 2: AUDIO + FUSION GATE                                    │
  │ ──────────────────────────────                                  │
  │ Freeze: text branch                                             │
  │ Train:  audio branch + quality gate + fusion gate + head        │
  │ LR:     5e-3, WD: 0.05                                         │
  │ Goal:   Audio learns to COMPLEMENT text (not override)          │
  │ Why:    With text frozen, audio must prove value through the    │
  │         fusion gate. If audio is useless → gate learns g→1     │
  │         (pure text) → no degradation.                           │
  ├─────────────────────────────────────────────────────────────────┤
  │ Stage 3: JOINT FINE-TUNING                                      │
  │ ─────────────────────────────                                   │
  │ Freeze: nothing                                                 │
  │ Train:  everything with reduced LR                              │
  │ LR:     1e-3, WD: 0.05                                         │
  │ Goal:   Polish fusion, improve CCC beyond stage 1 baseline.    │
  │ Why:    Both modalities are already well-initialized.           │
  │         Small joint adjustments can capture cross-modal signal. │
  └─────────────────────────────────────────────────────────────────┘

  REGULARIZATION:
    - Modality dropout: 15% audio dropout, 7.5% text dropout during training
    - Weight decay: 0.1 (stage 1), 0.05 (stages 2-3)
    - Gradient clipping: 1.0
    - Temporal dropout: 10% of chunks randomly dropped (20% for depressed)
    - Label noise: σ=0.15 for continuous PHQ-8 smoothing

8. VALIDATION METRICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Tracked per epoch during training:

  ┌─────────────────────┬──────────────────────────────────────────┐
  │ Metric              │ How computed                             │
  ├─────────────────────┼──────────────────────────────────────────┤
  │ CCC (overall)       │ Fusion output vs true PHQ-8              │
  │ CCC text-only       │ Zero audio → evaluate → CCC             │
  │ CCC audio-only      │ Zero text → evaluate → CCC              │
  │ Fusion gate mean    │ Average learned gate value g             │
  │ Loss                │ Weighted MSE on batch                    │
  └─────────────────────┴──────────────────────────────────────────┘

  SUCCESS CRITERIA:
    ✓ fusion_CCC ≥ text-only_CCC     (audio must not hurt)
    ✓ fusion_CCC ≥ 0.50              (target: match or beat text)
    ✓ audio_CCC > 0.0                (audio must have some signal)
    ✓ fusion_gate_mean ∈ [0.3, 0.8]  (both modalities contributing)

  All four metrics are logged every epoch. If fusion_CCC drops below
  text_CCC - 0.05 for 5+ epochs, a WARNING is logged.

9. FAILURE DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  AUTOMATIC DETECTION (in train_multimodal.py):

  1. PER-EPOCH: If fusion_CCC < text_CCC - 0.05:
     → Log WARNING with gate value
     → Indicates audio is actively hurting

  2. POST-TRAINING: If final fusion_CCC < stage1_CCC:
     → Log FAILURE_DETECT
     → Recommend using text-only model (stage 1 checkpoint)
     → Both checkpoints saved: best_stage1.pt (text fallback) and
       best_multimodal.pt (fusion)

  3. GATE COLLAPSE: If gate_mean → 1.0 consistently:
     → Audio branch is being ignored
     → Not a failure per se — system gracefully degrades to text-only
     → But signals audio features aren't useful

  4. FALLBACK MECHANISM:
     The model architecture INHERENTLY supports text-only fallback.
     If audio_features are all zeros (quality=0 or modality dropout):
     → quality gate zeros audio
     → fusion gate learns g→1
     → prediction comes from text branch only
     → CCC ≈ stage1 baseline (N loss degradation)

  ┌─────────────────────────────────────────────────────────────────┐
  │ FUSION STRATEGY                                                 │
  ├─────────────────────────────────────────────────────────────────┤
  │ 1. Quality gate: per-segment, suppresses bad audio              │
  │ 2. Each modality: StatsPool → BN → Project to 128-dim          │
  │ 3. Fusion gate: learned g = sigmoid(MLP([text, audio]))         │
  │ 4. Fused = g × text_proj + (1-g) × audio_proj                  │
  │ 5. Prediction head: Linear(128, 1) → PHQ-8 score               │
  │ 6. Modality dropout: 15% during training (text-only fallback)   │
